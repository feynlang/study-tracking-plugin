# Study Goal

모호한 학습 의도를 **반드시 산출물로 이어지는** 워크플로로 변환한다.
핵심 철학: **산출물 없이 "공부했다"라고 말하지 못하게 막는 것.**

## 진입점

이 스킬에 들어오는 세 가지 경로:

1. **온보딩/대시보드** — `/study-goal:start` 또는 처음 사용자의 모호한 인사
2. **새 학습 시작** — "X 공부하고 싶어" / URL 공유 / `/study-goal:new` / `/study-goal:link`
3. **기존 학습 액션** — 재개 / 일시중지 / 점검 / 샛길 질문 등

처음 진입 시 어느 경로인지 판단:
- `~/study-log/` 디렉토리 자체가 없음 → **온보딩 우선** (Step 0 건너뛰고 `/study-goal:start` 흐름으로)
- 디렉토리는 있지만 사용자 메시지가 모호 ("안녕", "시작", "사용법") → **대시보드** (`/study-goal:start` 흐름)
- 메시지에 명확한 학습 의도 → 바로 Step 0 → Step 1으로

## 두 가지 학습 모드

이 스킬은 입력을 보고 자동 분기:

- **일반 토픽 모드** — "트랜스포머 공부하고 싶어"
- **링크 모드** — "[URL]에 대해 공부하고 싶어"

URL 감지 규칙: 사용자 메시지에 `http://` 또는 `https://`로 시작하는
문자열이 있으면 링크 모드. 명시적 명령 `/study-goal:link <url>`로도 진입.

## 전체 흐름

```
Step 0: 활성 목표 확인 (다중 목표 관리)
Step 1: 목표 수집 + 모드 분기
Step 1.5: [링크 모드 전용] 콘텐츠 분석
Step 2: 개념 체크 (적응형 순차 질문)
Step 3: 종료 조건 설정 (4지선다)
Step 4: 산출물 형태 선택 (4지선다)
Step 5: 산출물 요구사항 구체화 (4지선다)
Step 6: artifact.md 생성 + 작업 시작
```

각 단계는 건너뛰지 않는다. Step 2는 사용자가 거부해도 최소 1개는 진행.

## 디렉토리 구조 (런타임)

```
~/study-log/
├── .active                       # 현재 활성 목표 slug (한 줄당 1개)
├── .claude-study-goal/
│   └── rubrics/                  # lazy 생성된 토픽 루브릭 누적
│       ├── ml-transformer.md
│       ├── link-github-karpathy-nanogpt.md
│       └── ...
├── <slug-1>/

│   ├── deliverable/              # 실제 산출물
│   └── _source/                  # [링크 모드, 옵션] 콘텐츠 스냅샷
└── <slug-2>/
    └── ...
```

## 상태 라이프사이클

각 학습 목표의 artifact.md는 다음 세 상태 중 하나:

```
🟡 진행 중   ← 새 목표 만들면 이 상태로 시작
   │
   ├── /study-goal:pause 또는 자연어("잠깐 멈출게") → 🔴 일시중지
   │                                                        │
   │                  /study-goal:resume 또는 자연어("이어서") ↓
   │                                                        │
   │  ←──────────────────────────────────────────────────────┘
   │
   └── 모든 체크 항목 [x] → 🟢 완료
```

상태 전이 규칙:
- `🟡 → 🔴`: `.active`에서 제거, 작업 로그에 일시중지 기록
- `🔴 → 🟡`: `.active`에 추가, 작업 로그에 재개 기록 (시간 차 포함)
- `🟡 → 🟢`: `.active`에서 제거, 회고 추가, 루브릭 다듬기 옵션 제안
- `🟢 → 🟡`: 일반적으로 막힘. 사용자가 명시적 요청 시에만 (드문 케이스)

## Step 0: 활성 목표 확인

### 행동
1. `~/study-log/.active` 파일 읽기 (없으면 빈 상태로 간주)
2. 활성 목표가 있으면 사용자 메시지가 어느 목표 컨텍스트인지 추론

### 새 목표 시작하려는 시그널 감지 시
사용자가 명백히 새 학습을 시작하려 함 ("X 공부하고 싶어", `/study-goal:new`):
- `.active`가 비어있으면 → 그대로 Step 1
- `.active`에 1개 있으면 → 사용자에게 묻기:
  - **옵션 A**: 기존 목표 일시중지하고 새 목표로 전환
  - **옵션 B**: 기존 목표 끝낸 뒤 시작
  - **옵션 C**: 둘 다 동시 진행 (multi-active)
- `.active`에 여러 개 있으면 (이미 multi-active) → 그냥 추가

### 기존 목표 재개 시그널 감지 시
다음 표현 감지 시 `/study-goal:resume` 흐름으로 매핑:
- "X 이어서 할게", "X 다시 시작", "X 계속할게"
- "어제 하던 거", "아까 하던 거", "지난번 하던 거"
- "다시 X로 돌아가자"

행동:
- 해당 목표의 상태가 `🔴 일시중지`였으면 `🟡 진행 중`으로 변경
- 해당 slug를 `.active`의 첫 줄로 이동 (없으면 추가)
- artifact.md 읽고 현재 상태 한 문단 요약해서 보여줌:
  - 종료 조건 / 진행률 / 마지막 작업 / 마지막 "막힌 점"
- 작업 로그에 재개 기록 append (시간 차 포함)
- "어디서부터 이어갈까요?" 후 사용자의 다음 액션 대기

### 일시중지 시그널 감지 시
다음 표현 감지 시 `/study-goal:pause` 흐름으로 매핑:
- "X 잠깐 멈출게", "X 일시중지", "X 나중에 할게"
- "오늘 여기까지" (현재 활성 목표 대상)

행동:
- 상태를 `🔴 일시중지`로 변경
- `.active`에서 해당 slug 제거
- 일시중지 사유 한 줄 묻기 (스킵 가능)
- 작업 로그에 일시중지 기록 append

### Multi-active 모드에서의 컨텍스트 추론
사용자 메시지가 어느 목표인지 애매하면 묻기:
- "이건 RAG 작업인가요, Rust 작업인가요?"
- 명확하면 그냥 그 목표 컨텍스트로 진행

## Step 1: 목표 수집 + 모드 분기

### 입력
사용자의 학습 의도

### 행동
1. URL 포함 여부 확인 → **모드 결정**
2. 일반 토픽 모드:
   - 토픽을 한 문장으로 재진술
   - topic slug 생성 (영문 소문자 + 하이픈, 예: `transformer-architecture`)
3. 링크 모드:
   - URL을 한 줄로 재진술 ("[제목]을 공부하고 싶으시군요")
   - link slug 생성 (예: `link-github-karpathy-nanogpt`, `link-blog-lilianweng-attention`)
4. `~/study-log/<slug>/` 존재 여부 확인 → 있으면 "이어서 / 새로" 묻기

### 출력
내부 상태: `{ mode: "topic" | "link", topic_or_url, slug, existing }`

### 다음 단계 트리거
- 일반 토픽 모드 → Step 2
- 링크 모드 → **Step 1.5**

---

## Step 1.5: [링크 모드 전용] 콘텐츠 분석

### 행동
1. `web_fetch`로 URL 페치
   - GitHub 레포면 README + (가능하면) 디렉토리 구조 + 핵심 파일 1-2개
   - 블로그/글이면 본문
   - 논문 PDF면 abstract + introduction + section 헤더
2. 페치 실패 시 → 사용자에게 핵심 내용 직접 입력 받기
3. 분석 결과 요약 제시:
   ```
   이 콘텐츠를 이렇게 이해했어요:
   - 종류: <GitHub 레포 / 블로그 / 논문>
   - 핵심 주제: <1-2문장>
   - 주요 구성: <레포면 디렉토리/파일, 글이면 섹션>
   - 전제 지식: <이걸 이해하려면 알아야 하는 것들>
   - 추정 학습 시간: <시간 범위>
   ```
4. 사용자에게 "맞나요? 수정/추가할 부분?" 확인

### 스냅샷 옵션
사용자에게 묻기: "원본을 로컬에 캡처해둘까요? (블로그처럼 사라질 수 있는
콘텐츠는 권장)"
- 예 → `~/study-log/<slug>/_source/`에 페치한 내용 저장
- 아니오 → URL과 페치 일시만 기록

### 출력
콘텐츠 분석 결과 (Step 2와 Step 5의 입력으로 사용)

---

## Step 2: 개념 체크 (적응형 순차 질문)

### 목적
사용자의 현재 이해도를 측정 → 이후 단계 산출물 난이도 결정

### 루브릭 로딩 절차 (Step 2 시작 전)
1. `~/study-log/.claude-study-goal/rubrics/<slug>.md` 찾기
2. 일반 토픽 모드: 토픽명 alias로도 검색 (예: "트랜스포머" → `ml-transformer.md`)
3. 매칭 없으면 → **루브릭 lazy 생성** (`templates/concept-check-rubrics/_generic.md` 참고)
   - Claude가 즉석에서 5단계 루브릭 작성
   - 사용자에게 보여주지 않고 즉시 저장
   - 링크 모드면 콘텐츠 분석 결과를 입력으로 사용
4. 루브릭의 샘플 질문은 **참고용** — 그대로 쓰지 않고 변형해서 사용

### 질문 규칙
- **1개씩 순차적으로** 질문 (한꺼번에 X)
- **최소 3개, 최대 5개**. 3개에서 수준 명확해지면 멈춤
- 첫 질문은 **L3 핵심** 난이도에서 시작
- 답변 평가 후 다음 질문 난이도 조절:
  - `correct` → 한 단계 어려운 개념
  - `partial` → 비슷한 수준의 다른 각도
  - `incorrect` 또는 `dont_know` → 한 단계 쉬운 개념
- 질문은 **서술형**. Yes/No, 4지선다 X
- 정답을 바로 알려주지 않음. "음, 그렇게 보시는군요" 정도로 받고 다음
- 링크 모드: 질문 소재는 **페치한 콘텐츠 안의 개념** 중심

### 내부 채점
각 질문마다 기록:
- 질문 / 답변 / 평가 (`correct` `partial` `incorrect` `dont_know`) / 난이도

### 종료 조건
- 5개 완료 / 3개 후 명확 / 사용자가 "그만" 명시 (단 최소 1개는 했어야 함)

### Step 2.5: 요약 + 피드백
체크 끝난 후 한꺼번에:
1. 각 질문의 정답/보완 설명 (짧게)
2. 추정 수준: `초급` / `중급` / `상급`
3. 보완 필요한 개념 목록
4. 다음 단계 안내

### 학습 종료 후 루브릭 다듬기 (옵션)
모든 체크 항목이 `[x]`가 되어 학습이 끝나면, Claude는 한 번 제안:
- "이번 학습 결과를 토대로 이 토픽 루브릭을 다듬을까요?"
- 동의하면 기존 루브릭 파일을 업데이트 (어떤 개념이 실제로 핵심이었는지 등)

---

## Step 3: 종료 조건 설정 (4지선다)

### 행동
목표 + 개념 체크 수준 기반으로 **정확히 4가지** 옵션 생성.

#### 일반 토픽 모드의 4가지 차원
- **시간 기반**: 총 N시간 투자
- **양 기반**: N개 개념 / N개 문제 / N줄 코드
- **능력 기반**: X를 혼자서 Y 없이 할 수 있음
- **외부 검증**: X에게 설명해서 통과 / X 테스트 통과

#### 링크 모드의 4가지 차원
- **재현형**: 이 코드/글을 보지 않고 다시 작성
- **확장형**: 이 콘텐츠에 X 기능/관점 추가 (구체적 X 명시)
- **분석형**: 모든 결정/주장에 "왜 이렇게 했는가" 답할 수 있음
- **비판형**: 이 콘텐츠의 한계/개선점 N개 제시

각 옵션은 1-2문장으로 구체적으로. "충분히 이해" 같은 모호한 표현 금지.

`AskUserQuestion`으로 제시.

### 출력
선택된 종료 조건 1개

---

## Step 4: 산출물 형태 선택 (4지선다)

### 고정 옵션 (모드 무관)
- **프로젝트** — 작동하는 코드/시스템 1개
- **정리된 노트** — 학습 자료 문서
- **문제 풀이** — N개 연습 문제 풀이
- **설명 자료** — 남에게 가르치는 발표/글

### 추천 마크 규칙
개념 체크 수준에 따라 `(추천)` 표시:
- 초급 → "노트" 또는 "문제 풀이"
- 중급/상급 → "프로젝트" 또는 "설명 자료"

링크 모드의 경우 추가 추천:
- GitHub 레포 → "프로젝트"
- 블로그/논문 → "정리된 노트" 또는 "설명 자료"

추천은 강제 아님.

---

## Step 5: 산출물 요구사항 구체화 (4지선다)

### 행동
**목표 + 개념 체크 수준 + 종료 조건 + 산출물 형태**(+ 링크 모드면 콘텐츠 분석)
모두 반영해 **정확히 4가지** 변형 생성.

참고 템플릿:
- 일반 토픽 모드: `templates/deliverables/<form>.md`
- 링크 모드: `templates/deliverables-link/<content-type>.md`

각 변형은:
- 이름 (한 줄)
- 무엇을 만드는지 (1-2문장)
- 체크가능한 완료 항목 3-5개
- 예상 소요 시간

### 출력
선택된 변형 + 완료 체크 항목

---

## Step 6: artifact.md 생성 + 작업 시작

### 행동
1. `~/study-log/<slug>/artifact.md` 생성
2. `~/study-log/<slug>/deliverable/` 빈 디렉토리 생성
3. 링크 모드 + 스냅샷 동의했으면 `~/study-log/<slug>/_source/` 생성
4. `.active`에 slug 추가 (multi-active면 append, single이면 덮어쓰기)
5. artifact.md 채움 (아래 템플릿)
6. 사용자에게 요약 + 작업 시작 의사 확인

### artifact.md 템플릿

```markdown
# <topic 또는 콘텐츠 제목>

**시작일**: <YYYY-MM-DD>
**모드**: <일반 토픽 / 링크>
**상태**: 🟡 진행 중

---

## 📌 목표 (불변)

<Step 1에서 재진술한 한 문장>

---

[링크 모드일 때만 이 섹션 추가]

**URL**: <원본 URL>
**종류**: <GitHub 레포 / 블로그 / 논문 / 기타>
**캡처 일시**: <YYYY-MM-DD HH:MM>
**핵심 요약**: <1-3문장>
**주요 구성**:
- ...
```

## 작업 로그 append 규칙

### 1. Hook 자동 로그 (`deliverable/` 내 파일 변경 시)
hook이 자동으로:
```
- YYYY-MM-DD HH:MM — `Write` on `deliverable/attention.py`
```

### 2. Claude의 턴 단위 요약
"의미 있는 진전"이 있을 때 턴 마지막에 append:
```
### YYYY-MM-DD HH:MM
**무엇을 했나**: <1-3문장>
**다음 스텝**: <1-2문장>
**막힌 점** (있다면): <1문장>
```

"의미 있는 진전" 기준:
- 코드 파일 생성/주요 수정
- 개념 설명/이해 대화가 길었음
- 체크리스트 항목 1개 완료
- 사용자가 "오늘 여기까지" 명시

### 3. 모든 체크 항목 완료 시
- 상태를 `🟢 완료`로 변경
- 최종 회고 1문단을 작업 로그에 추가
- 루브릭 다듬기 옵션 제안 (Step 2.5 참조)
- `.active`에서 이 slug 제거
- 축하 메시지

## 샛길 질문 처리

### 진입 트리거
- 사용자: "잠깐 질문 하나만", "샛길로 빠지자면", "아 근데 X는 뭐야?"
- 명시적 명령: `/study-goal:sidebar`
- Claude가 감지 시 확인: "샛길 질문으로 받을게요, 본 작업 돌아갈 때 알려주세요"

### 답변 깊이 조절
- **기본**: 3-5문장 핵심 답변
- 사용자가 "더 자세히" 1번 → 1-2문단 추가
- 사용자가 "더" 2-3번 째 → 한 번 묻기:
  - "이 주제 자체로 별도 학습 목표를 만들까요? 지금 [메인 목표] 작업이 멈춰있어요."
  - 옵션:
    - ① 핵심만 듣고 돌아가기
    - ② 새 학습 목표로 분리 (Step 1부터, 단 개념 체크 일부 상속 가능)
    - ③ 일단 더 듣기 (다시 한번만)

### 기록 방식
artifact.md의 "🔍 샛길 질문" 섹션에 append:
```markdown
### YYYY-MM-DD HH:MM
**질문**: <원래 질문 그대로>
**맥락**: <메인 작업 중 어디에서 나왔는지 1줄>
**핵심 답변**: <3-5문장 요약>
**본 학습과의 관계**: <어떻게 연결되는지 1줄>
**더 파볼 가치**: ★★☆ (낮음 / 중간 / 높음)
```

### 별도 학습 목표로 승격 시
- 새 목표 Step 1 진입 (multi-active 모드로)
- 본 목표 artifact.md에 표시:
  ```
  → 이 샛길 질문은 [새 slug]로 분리됨
  ```

### 이탈
- 사용자: "본 작업으로", "원래 하던 거 계속"
- Claude가 충분히 답한 뒤 자동 제안: "샛길 답변 끝났어요. 본 작업 돌아갈까요?"

## 예외 처리

### "개념 체크 건너뛰자"
허용하지 않음. 최소 1개 진행. "정말 초보예요" 답변도 유효한 데이터.

### "그냥 바로 만들게"
허용하지 않음. "산출물을 내도록 강제하는 게 핵심이에요. 최소 Step 3-5만이라도
같이 정해요. 빠르게 갈 수 있어요."로 유도.

### "종료 조건 바꾸고 싶어"
허용. 불변 섹션 직접 수정 + 변경 주석 추가 + 작업 로그에도 변경 사실 기록.

### URL 페치 실패 (링크 모드)
- 사용자에게 콘텐츠 핵심 직접 입력 요청
- 입력받은 내용으로 Step 1.5 분석 진행

### 다중 목표 충돌 시
같은 슬러그로 새 목표 시작하려 하면:
- "이 슬러그가 이미 있어요. ① 이어서 / ② 다른 이름으로 / ③ 기존 폐기 후 새로"

- 사용자 언어 따라가기 (한국어 기본)

- 샛길 질문은 환영 (호기심 죽이면 학습 죽음). 단 깊이 제한.
