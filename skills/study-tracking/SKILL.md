name: study-tracking
description: Turn a vague learning goal into a concrete deliverable with clear completion criteria, then enforce actually producing that deliverable. Use this skill whenever the user expresses a learning intent — phrases like "공부하고 싶어", "배우고 싶어", "I want to learn X", "study X", "마스터하고 싶어", or describes wanting to understand a topic without a clear plan. This skill pushes the user through concept check → completion criteria → deliverable requirements → actual production, and auto-logs the process to artifact.md.

# Study Tracking

모호한 학습 의도를 **반드시 산출물로 이어지는** 워크플로로 변환한다.
핵심 철학: **산출물 없이 "공부했다"라고 말하지 못하게 막는 것.**

## 전체 흐름 (6단계)

```
Step 0: 첫 실행 감지 (온보딩)
Step 1: 목표 수집
Step 2: 개념 체크 (적응형 순차 질문)
Step 3: 종료 조건 설정 (4지선다)
Step 4: 산출물 형태 선택 (4지선다)
Step 5: 산출물 요구사항 구체화 (4지선다)
Step 6: artifact.md 생성 + 작업 시작
```

각 단계는 건너뛰지 않는다. 특히 Step 2는 사용자가 "그냥 넘어가자"고
해도 **최소 1개 질문은 한다** — 이 스킬의 존재 이유가 사용자가 자기
수준을 직면하게 만드는 것이기 때문.

---

## Step 0: 첫 실행 감지 (온보딩)

### 조건
`~/study-log/` 디렉토리가 존재하지 않을 때만 실행.
이미 존재하면 Step 0 전체를 건너뛰고 Step 1로 바로 진행.

### Claude의 행동
1. `~/study-log/` 디렉토리 생성
2. 아래 온보딩 메시지를 출력

### 온보딩 메시지 형식

```
안녕하세요! **study-tracking** 플러그인이에요 👋

"공부해야지"에서 끝나는 게 아니라, **실제로 만든 것**으로 끝나도록 도와줘요.

매번 이렇게 진행해요:
  1. 뭘 배우고 싶은지 말해주면
  2. 현재 수준을 같이 확인하고
  3. 종료 조건 + 산출물 형태를 정하고
  4. 작업하는 동안 진행 상황을 자동으로 기록해요

**명령어:**
  /study-tracking:new   → 새 학습 목표 시작
  /study-tracking:check → 진행 중인 목표 점검

학습 로그는 ~/study-log/ 에 저장돼요. 방금 만들었어요.

---
자, 뭘 공부하고 싶으세요?
```

### 다음 단계 트리거
메시지 출력 후 사용자 응답을 기다린다. 응답이 오면 Step 1로 진행.

---

## Step 1: 목표 수집

### 입력
사용자의 학습 의도 ("트랜스포머 공부하고 싶어" 등)

### Claude의 행동
1. 목표를 한 문장으로 재진술 — 오해 방지
2. topic slug 생성 (영문 소문자 + 하이픈, 예: `transformer-architecture`)
   - 사용자에게 보여주지 않고 내부적으로만 사용
3. `~/study-log/<slug>/` 디렉토리가 이미 있는지 확인
   - 있으면: "이 주제로 이전에 시작한 학습이 있어요. 이어서 할까요, 새로 시작할까요?"
   - 없으면: 다음 단계로

### 출력
내부 상태: `{ topic, slug, existing }`

### 다음 단계 트리거
사용자가 재진술에 동의 (또는 수정 후 동의)

---

## Step 2: 개념 체크 (적응형 순차 질문)

### 목적
사용자의 현재 이해도를 **측정**하고, 이 측정값이 이후 단계의
산출물 난이도를 결정하는 데 쓰인다.

### 규칙
- **1개씩 순차적으로** 질문. 여러 개 한꺼번에 묻지 않음.
- **최소 3개, 최대 5개** 질문. 3개에서 명확히 수준 파악되면 멈춤.
- 첫 질문은 **중간 난이도**에서 시작 (토픽의 핵심 개념).
- 답변 평가 후 다음 질문 난이도 조절:
  - 정확하게 답함 → 한 단계 어려운 개념
  - 부분적으로 답함 → 비슷한 수준의 다른 각도 질문
  - 모른다고 함 / 틀림 → 한 단계 쉬운 개념 (또는 전제 개념)
- 질문 형식: **서술형** (4지선다 아님). "X가 뭔가요?", "왜 Y가 필요한가요?" 같은.
- 사용자 답변에 대해 **정답을 바로 알려주지 않음**. "음, 그렇게 보시는군요" 정도로 받고 다음 질문. 개념 체크가 끝난 후 Step 2.5에서 한꺼번에 피드백.

### Claude의 내부 채점
각 질문마다 내부적으로 기록:
- 질문 내용
- 사용자 답변
- 평가: `correct` / `partial` / `incorrect` / `dont_know`
- 난이도: `L1 전제` / `L2 기초` / `L3 핵심` / `L4 응용` / `L5 심화`

### 종료 조건
다음 중 하나:
- 5개 질문 완료
- 3개 질문 후 수준이 명확 (예: 3개 모두 correct → 상급, 3개 모두 incorrect → 초급)
- 사용자가 "이제 그만"이라고 명시적으로 요청 (최소 1개는 이미 했어야 함)

### Step 2.5: 개념 체크 요약
체크 후 Claude는 한 번에:
1. 각 질문의 정답/보완 설명 (짧게)
2. 추정된 수준: `초급 / 중급 / 상급` 중 하나
3. 다음 단계 제안: "이 수준이면 [X 형태]의 산출물이 적절할 것 같아요"

### 출력
```
{
  concept_check_results: [
    { q: "...", a: "...", eval: "partial", level: "L3" },
    ...
  ],
  estimated_level: "중급"
}
```

---

## Step 3: 종료 조건 설정 (4지선다)

### 목적
"얼마나 하면 끝난 건지"를 산출물 형태 정하기 **전에** 먼저 정한다.
이게 뒤집혀 있으면 산출물에 맞춰 종료 조건이 늘어짐.

### Claude의 행동
목표 + 개념 체크 수준을 바탕으로 **정확히 4가지** 종료 조건 옵션 생성.
각 옵션은 서로 다른 차원으로 구성:

- 옵션 A: **시간 기반** — "총 N시간 투자"
- 옵션 B: **양 기반** — "N개 개념 / N개 문제 / N줄 코드"
- 옵션 C: **능력 기반** — "X를 혼자서 Y 없이 할 수 있음"
- 옵션 D: **외부 검증 기반** — "X에게 설명해서 통과" / "X 테스트 통과"

각 옵션은 **1-2문장으로 구체적으로**. "충분히 이해" 같은 모호한
표현 금지.

`ask_user_input_v0` (Claude Code에선 `AskUserQuestion`)으로 제시.

### 출력
선택된 종료 조건 1개.

---

## Step 4: 산출물 형태 선택 (4지선다)

### 고정 옵션
항상 이 4개:
- **프로젝트** — 작동하는 코드/시스템 1개
- **정리된 노트** — 학습 자료 문서
- **문제 풀이** — N개 연습 문제 풀이
- **설명 자료** — 남에게 가르치는 발표/글

### Claude의 행동
단순 4지선다 제시. 개념 체크에서 추정된 수준에 따라 **추천 마크**만 추가:
- 초급 → "노트" 또는 "문제 풀이" 옆에 `(추천)`
- 중급 → "프로젝트" 또는 "설명 자료" 옆에 `(추천)`
- 상급 → "프로젝트" 또는 "설명 자료" 옆에 `(추천)`

추천은 강제하지 않음. 사용자 선택 존중.

### 출력
선택된 산출물 형태 1개.

---

## Step 5: 산출물 요구사항 구체화 (4지선다)

### 목적
"프로젝트"만으로는 너무 광범위. 구체적인 스펙을 고르게 한다.

### Claude의 행동
**목표 + 개념 체크 수준 + 종료 조건 + 산출물 형태** 네 가지를 모두 반영해
**정확히 4가지** 구체적 요구사항 변형 생성.

참고: `templates/<deliverable_type>.md`의 변형 생성 가이드라인.

각 변형은:
- 이름 (한 줄, 구체적)
- 무엇을 만드는지 (1-2문장)
- 체크가능한 완료 항목 3-5개
- 예상 소요 시간

예시 (목표: "트랜스포머 이해", 수준: 중급, 종료조건: "GPT-2를 처음부터 만들기", 형태: 프로젝트):

1. **Nano-GPT 재현** — karpathy nanoGPT를 참고해 직접 타이핑, 셰익스피어 학습
2. **Toy Attention 시각화** — multi-head attention을 PyTorch로 구현 + attention weight 시각화
3. **Scratch 트랜스포머** — 참고 코드 없이 논문만 보고 encoder-only 트랜스포머 구현
4. **Transformer 블록 라이브러리** — 재사용 가능한 트랜스포머 블록을 여러 개 구현 + 테스트

`ask_user_input_v0`으로 제시.

### 출력
선택된 요구사항 스펙. 완료 체크 항목 포함.

---

## Step 6: artifact.md 생성 + 작업 시작

### Claude의 행동
1. `~/study-log/<slug>/artifact.md` 생성
2. `~/study-log/<slug>/deliverable/` 빈 디렉토리 생성
3. artifact.md를 아래 템플릿으로 채움
4. 사용자에게 요약 보여주고 작업 시작 의사 확인

### artifact.md 템플릿

```markdown
# <topic>

**시작일**: <YYYY-MM-DD>
**상태**: 🟡 진행 중

---

## 📌 목표 (불변)

<Step 1에서 재진술한 한 문장>

---

## 🧠 개념 체크 결과 (불변)

**추정 수준**: <초급/중급/상급>

| # | 질문 | 답변 요약 | 평가 | 난이도 |
|---|------|-----------|------|--------|
| 1 | ... | ...       | ...  | L3     |
| 2 | ... | ...       | ...  | L4     |
| 3 | ... | ...       | ...  | L3     |

**보완이 필요한 개념**:
- ...
- ...

---

## 🏁 종료 조건 (불변)

<Step 3에서 선택된 것, 원문 그대로>

---

## 📦 산출물 요구사항 (불변)

**형태**: <Step 4에서 선택>
**스펙**: <Step 5에서 선택한 변형의 이름>

**설명**: <Step 5 변형 설명 1-2문장>

**완료 체크 항목**:
- [ ] <항목 1>
- [ ] <항목 2>
- [ ] <항목 3>
...

**예상 소요 시간**: <...>

---

## 📝 작업 로그 (append-only)

<!-- 이 아래는 hook과 Claude가 자동으로 추가함. 수동 편집 금지. -->

```

### 중요: "불변" 섹션은 덮어쓰지 않음
Step 6 이후로 Claude는 **"작업 로그" 섹션 아래에만** append.
위 불변 섹션을 수정하려면 사용자가 명시적으로 요청해야 함.

### 출력
artifact.md 경로를 사용자에게 알리고, "이제 시작하자"로 턴 종료.

---

## 작업 로그 append 규칙

세션이 진행되는 동안, artifact.md의 "작업 로그" 섹션 아래에
다음 시점에 항목이 추가됨:

### 1. Hook 자동 로그 (`deliverable/` 내 파일 변경 시)
hook이 자동으로 다음 형식 추가:
```
- YYYY-MM-DD HH:MM — `Write` on `deliverable/attention.py`
```

### 2. Claude의 턴 단위 요약 (사용자가 의미 있는 작업을 했을 때)
Claude는 한 턴에서 의미 있는 진전이 있었다면, 턴 마지막에 artifact.md에
다음 형식 추가:
```
### YYYY-MM-DD HH:MM
**무엇을 했나**: <1-3문장>
**다음 스텝**: <1-2문장>
**막힌 점** (있다면): <1문장>
```

"의미 있는 진전"의 기준:
- 코드 파일 생성/주요 수정
- 개념 설명/이해 대화가 길었음
- 체크리스트 항목 1개 완료
- 사용자가 명시적으로 "오늘 여기까지"라고 말함

### 3. 체크 항목 완료 표시
사용자가 체크리스트 항목을 완료했다고 명시하면, Claude는
**불변 섹션의 체크박스를 `- [x]`로 변경**. 이건 예외적으로 불변
섹션을 건드리는 유일한 경우.

모든 체크 항목이 `[x]`가 되면:
- 상태를 `🟢 완료`로 변경
- 최종 회고 1문단을 작업 로그에 추가
- "축하합니다, 이 학습은 끝났어요" 메시지

---

## 예외 처리

### "개념 체크 건너뛰자"
허용하지 않음. 최소 1개는 하되, "정말 초보예요/전혀 모르겠어요"라고
답해도 그 자체로 유효한 데이터. "알겠습니다, 초급으로 진행할게요"로
넘어감.

### "그냥 바로 만들게"
허용하지 않음. "산출물을 내도록 강제하는 게 이 워크플로의 핵심이라서,
최소한 Step 3-5(종료 조건, 형태, 스펙)는 같이 정하고 시작할게요.
빠르게 갈 수 있어요"로 유도.

### "종료 조건을 바꾸고 싶어"
작업 로그 섹션에 변경 기록 남기고 허용. 불변 섹션을 직접 수정하되,
변경 날짜 + 이유 주석 추가:
```markdown
## 🏁 종료 조건 (불변)

<!-- 2026-04-25 수정: 범위가 너무 넓어서 축소 -->
<새 종료 조건>
```

---

## 스타일

- 사용자 언어 따라가기 (한국어 기본)
- 한 번에 한 질문 (Step 2에서 특히 중요)
- 각 Step이 끝날 때마다 "다음은 X 단계예요" 같은 안내 짧게 제공
- 사용자가 흐름을 깨려고 하면 부드럽게 되돌림, 강요 아님
